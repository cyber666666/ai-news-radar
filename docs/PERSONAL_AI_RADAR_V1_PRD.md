# 个人 AI 行业资讯雷达 V1 产品需求文档

## 1. 项目背景

本项目基于开源项目 `LearnPrompt/ai-news-radar` 进行轻量化改造，服务一名准备算法岗位、重点关注推荐系统的 2027 届毕业生。

长期关注方向：

- 推荐、搜索与广告算法；
- 大模型与多模态模型；
- AI Agent；
- AI Infra；
- AI 产品与 Vibe Coding；
- 国内外 AI 从业者、研究者和产品负责人的观点；
- 具有业务理解和工业落地价值的内容。

目标不是开发复杂推荐系统或堆叠功能，而是构建一个可长期稳定使用的个人 AI 资讯阅读网页。

## 2. 产品目标

建立一个自动更新的个人 AI 行业资讯网页：

- 保留原项目高质量官方、媒体和社区来源；
- 支持个人博客、RSS、X 作者、小红书博主和公众号；
- 自动抓取、去重、分类并展示最近资讯；
- GitHub Actions 按固定频率更新；
- 用户每天早上可直接打开网页阅读；
- 单个不稳定来源失败不得影响其他来源、数据生成或网页更新；
- 后续根据真实使用体验逐步增加来源和功能。

## 3. 非目标

V1 不实现：

- 移动 App、注册登录或多用户系统；
- 推荐模型训练、向量数据库、复杂用户画像或点击率预测；
- 多 Agent 工作流；
- 整体前端重写；
- 非现有运行所必需的数据库；
- 所有小红书博主和公众号的全自动抓取；
- 为部署优先引入 Docker；
- 无必要的大规模代码重构。

## 4. 目标用户与场景

V1 仅服务项目维护者本人。主要场景：

- 每天早上浏览最近 24 小时新增内容；
- 优先查看推荐系统、大模型、AI 产品和 AI Infra；
- 点击原文深入阅读；
- 白天偶尔查看最新更新；
- 根据长期阅读体验调整来源。

## 5. V1 内容范围

### 5.1 保留现有来源

默认保留：

- 官方 AI 公司动态；
- AI 媒体；
- Hacker News；
- GitHub 与开发者工具更新；
- AI 社区与聚合来源；
- Follow Builders 等公开 Feed；
- 当前默认 OPML 示例来源。

除非确认来源长期低质量、重复严重或持续抓取失败，否则 V1 不删除原有来源。

### 5.2 个人重点来源

1. 推荐、搜索和广告：工业实践、召回/粗排/精排/重排、CTR/CVR、生成式推荐、多任务学习、行为序列建模和工程优化。
2. 大模型与新模型：模型发布、开源、多模态、Agent 模型、评测、SOTA、训练与推理方法。
3. AI Infra：分布式训练、推理加速、GPU、Serving、Agent 基础设施、向量检索、数据基础设施与部署实践。
4. AI 产品与 Vibe Coding：产品设计、Agent/Coding Agent、AI 创业、产品负责人观点、需求、商业模式与增长。
5. 个人关注作者：独立博客、X、小红书、公众号、Newsletter，以及国内外研究者、工程师和产品负责人。

## 6. 信息源接入策略

按以下优先级逐个评估来源：

1. 官方 RSS/Atom：存在稳定 Feed 时直接加入个人 OPML。
2. 公开 JSON、GitHub Feed 或官方 API：优先使用现有抓取器，必要时增加轻量抓取函数。
3. 公开静态页面：只接无需登录、结构相对稳定的页面；适配器必须具备超时、错误处理、时间解析、URL 去重和单源数量限制。
4. 密钥型第三方 API：密钥只存在 GitHub Secrets 或本地环境变量；设置数量、频率与费用上限；失败不得中断主流程。
5. 手动收录：没有稳定自动化方式时保留手动链接方案。V1 宁可半自动，也不实现脆弱的复杂爬虫。

## 7. 个人来源清单

创建 `docs/MY_SOURCE_INVENTORY.md`，每个来源记录：

| 字段 | 说明 |
| --- | --- |
| 名称 | 博主、公众号、网站或组织名称 |
| 平台 | RSS、博客、X、小红书、公众号等 |
| 主页地址 | 原始主页 |
| 重要程度 | P0、P1、P2 |
| 关注原因 | 长期关注原因 |
| 预计接入方式 | RSS、API、静态页面、手动链接等 |
| 是否需要密钥 | 是或否 |
| 当前状态 | 待评估、已接入、测试中、跳过 |
| 最近测试时间 | 最近一次抓取测试时间 |
| 备注 | 风险、频率和维护说明 |

优先级定义：P0 必须关注（不超过 10 个）；P1 重要来源（20–30 个）；P2 观察来源（暂不自动接入或低频更新）。

## 8. 产品功能需求

### 8.1 自动抓取

- GitHub Actions 每 4 小时运行一次；
- 计划时刻为 00:00、04:00、08:00、12:00、16:00、20:00；
- 工作流明确 GitHub cron 使用 UTC，并在运维文档标注对应北京时间；
- 页面展示实际生成时间。

### 8.2 最近内容

- 首页展示最近 24 小时，按发布时间倒序；
- 保留“精选/全量”、现有栏目、关键词搜索和来源搜索；
- V1 不改造成复杂个性化推荐首页。

### 8.3 内容字段

每条资讯至少保留标题、中文标题/翻译、来源、作者、原文链接、发布时间、类别、来源类型、摘要/推荐理由、是否官方一手、抓取时间。字段缺失允许为空，不得因此丢弃整条内容。

### 8.4 去重

- 相同 URL 去重；
- 去除跟踪参数后去重；
- 标题高度相似内容去重；
- 同一事件继续使用原项目故事合并机制，并保留多来源展开。

### 8.5 来源健康状态

沿用 `data/source-status.json`，至少记录成功状态、条目数、耗时、错误原因、RSS 失败数量、高级来源是否跳过和最近成功时间。

### 8.6 故障隔离

单个 RSS、小红书、X API 或公众号失败不得阻塞其他来源、数据生成或 GitHub Pages 发布；配额耗尽时自动跳过；错误进入来源状态文件。

## 9. 个性化要求

调整个人 persona，重点关注：

- 推荐、搜索和广告算法；
- 大模型和多模态；
- AI Infra 与工程优化；
- AI Agent；
- AI 产品与 Vibe Coding；
- 工业落地、业务理解、算法求职和项目学习；
- 国内外高质量作者的原创观点。

降低纯营销软文、无技术/产品细节的融资新闻、重复转载、标题党、娱乐化 AI、无方法或来源说明的榜单、缺少原始出处的二手信息。

V1 复用现有“全部、模型、产品、开发者、行业、论文、社区、自媒体”栏目，可补充推荐系统、搜索、广告、AI Infra、Agent、Vibe Coding、求职相关标签，但不重构分类系统。

## 10. 小红书接入

首批仅评估 3–5 个 P0 博主。先确认是否有稳定 API/作者接口；如果只有关键词搜索，需验证作者覆盖准确性；设置每日上限并默认每天运行一次；不提交 Cookie 或登录凭据。

自动抓取不可用时标记为 `manual`，允许手动收录链接，不阻塞 V1 发布。全量自动监控不是 V1 验收条件。

## 11. 微信公众号接入

逐账号检查官方 RSS、公开文章列表、合法稳定第三方 Feed 或手动链接路径。优先 3–5 个 P0 公众号。不提交私人 Cookie、不抓取私人消息、不绕过访问控制，失败不得阻塞主流程。全量自动获取不是 V1 验收条件。

## 12. 部署方案

正式部署：GitHub Actions 负责抓取与生成数据，GitHub Pages 发布静态网页，GitHub Secrets 保存 API Key 和私人 OPML，仓库 `data/` 保存公开 JSON。

本地仅用于调试、来源测试、单元测试、页面检查和 Actions 故障排查，不要求长期运行。V1 不新增 Docker；只有需要持久登录、Playwright、Cookie、数据库或 GitHub Actions 无法稳定承担的社交平台采集时，才评估独立采集服务。

## 13. 安全与隐私

禁止提交 API Key、Token、Cookie、会话、私人邮箱正文、私人 OPML、浏览器配置、`.env` 或个人敏感信息。所有密钥使用 GitHub Actions Secrets 或本地环境变量；`.env.example` 仅保留变量名。

需检查 `.gitignore`、Git 历史、Actions 日志、生成 JSON 和错误日志；错误不得打印完整 API Key、Cookie 或 Authorization Header。

## 14. 实施阶段

1. 仓库审计：阅读现有文档与工作流，输出架构、入口、产物和部署流程，不立即重构。
2. 原版跑通：安装依赖，完整抓取，启动静态网页，运行测试，只修复阻塞问题。
3. 线上基线：启用 Actions、手动运行、确认 `data/*.json` 更新、启用 Pages 并检查页面时间。
4. 更新频率：改为每 4 小时，保留 `workflow_dispatch`，检查 UTC、并发和日志可诊断性。
5. 个人 RSS：建立清单和 OPML，通过 `FOLLOW_OPML_B64` 保存私人 OPML，检查来源状态。
6. 个人兴趣：调整 persona，保留前端，比较调整前后输出。
7. 小红书、X 和公众号：先评估，后选择 3–5 个高优先级来源实现；所有高级源故障隔离且有预算上限。
8. 连续运行验收：至少运行 7 天，记录更新、数量、重复、高质量比例、失败源、阅读体验、遗漏和噪声。

## 15. 验收标准

- GitHub Pages 可访问；
- GitHub Actions 可手动成功运行并每 4 小时自动运行；
- 最近 24 小时内容正常显示；
- 至少新增 10 个个人 RSS 或公开稳定来源，至少一个出现在网页；
- `source-status.json` 显示来源状态；
- 单源失败不影响整体；
- 仓库及 Git 历史无密钥、Cookie、Token；
- 手机和桌面均可阅读；
- 连续运行 7 天无严重阻塞；
- 小红书与公众号自动化不作为 V1 强制验收条件。

## 16. 最终交付物

- 可访问的 GitHub Pages；
- 每 4 小时运行的 GitHub Actions；
- 个人 OPML 与 persona；
- `docs/MY_SOURCE_INVENTORY.md`；
- `docs/DEPLOYMENT.md`；
- `docs/OPERATIONS.md`；
- `.env.example`；
- 来源健康状态文件；
- 7 天运行观察记录；
- V1.1 待办清单。

## 17. 开发原则

先跑通再个性化；先稳定来源再社交平台；先配置后改代码；不为开发而开发；不重写已有能力；不提前引入数据库或 Docker；每阶段独立可运行；修改后运行测试；密钥与代码分离；优先保证长期可维护和真实阅读体验。
